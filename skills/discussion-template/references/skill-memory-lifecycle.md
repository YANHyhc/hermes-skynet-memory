# Skill 记忆生命周期设计原则

> **原则：Skill 应自带"记忆分层"——让信息在正确的时间出现在正确的位置，过期自动退场。**
> 源自2026-07-10晏翔讨论固化

## 核心模型

```
Skill执行 → 分级匹配 → 完成后降级 → 最终只留提醒
                                    ↓
                             长期归档不再干扰
```

## 三层记忆映射

| 层级 | 范围 | Skill适用场景 | 数据源 | 噪声 |
|:----|:-----|:-------------|:------|:----:|
| **短期** | 当日 | 首次执行·新任务·快速应答 | 实时 state.db | 极少 |
| **中期** | 本周~跨周 | 流程成熟·周期性执行·有修正记录 | 实时 state.db | 适中 |
| **长期** | 历史全量 | 策略复盘·异常分析·跨月参考 | 管道库 (pipeline/state_pipeline.db) | 全量但被动 |

## 生命周期示例：竞彩结算

```
第1次：短期   → 查当日规则，轻量执行
第2-7次：中期 → 查近期结算记录，参考修正记录
稳定后：长期   → 只有"今日提醒"，历史归档不干扰
                需查历史时用户主动触发
```

## 何时查什么

| 条件 | 查哪层 |
|:----|:------|
| 首次执行某skill | 短期 |
| 该任务已执行但未跨周 | 中期 |
| 任务跨周/用户提到"历史""之前" | 长期（管道库） |
| 任务已稳定运行超过一个月 | 只留当日提醒，不主动查历史 |

## Skill 编写指南

每个 skill 的 frontmatter 应声明其预期记忆层级：

```yaml
memory_tier: short-term | medium-term | long-term | auto
# short-term: 一次性任务，用完即弃
# medium-term: 周期性任务，保留近期上下文
# long-term: 参考型任务，需要全量历史
# auto: 按执行次数自动降级
```

## 降级规则

```
短期 → 执行3次后无修改 → 自动转为中期
中期 → 跨周且无异常 → 自动转为长期提醒
长期 → 任何异常/用户修改 → 回退至中期（重新关注）
```

## 与此原则冲突时的判断

当用户要求执行某个 skill，但其记忆层级不匹配时：
- **短期任务不要查历史** → 快
- **中期任务不要扫全量** → 准
- **长期任务不要占实时库** → 稳
