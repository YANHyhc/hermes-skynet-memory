# 讨论固化框架 v1.4

> 🆕 v1.4 (2026-07-11): 新增H015审计前全证据验证·五维审计框架（逻辑自洽/可行有效/效率提升/断链解决/死循环风险）·memory-auto-tier实施计划关联

## 🚨 硬规则：「讨论:」前缀 = 无条件触发

**用户消息以"讨论:"、"讨论模式"、"讨论系统"、"讨论skill"开头 → 立即无条件进入讨论模式。**

无论当前在做什么、进行到什么程度、处于什么上下文——立刻停止当前工作流，强制切换至讨论流程。

不做例外判断："但这个话题之前讨论过"/"这个只是顺便问一下"/"我先回答完这个问题"——全都不允许。

## 使用方式

当用户说"讨论 XXX"、"讨论:"或"继续 XXX 的讨论"时：

1. 先查 `.discussions/{module}/INDEX.md` 看进度
2. 加载对应文件 `.discussions/{module}/YYYY-MM-DD.md` 恢复上下文
3. 按5步流程继续讨论
4. 每次讨论结束后更新文件

**始终只讨论不执行** — 除非用户明确说"干了"、"实施"或"可以了，执行"。

## 5步讨论流程

| 步骤 | 内容 | 确认标志 |
|:---:|------|:--------:|
| ① 需求定义 | 解决什么问题？谁用？期望产出是什么？ | 用户说"对"/"是的" |
| ② 输入输出 | 需要什么数据？产出什么结果？格式？ | 用户确认边界 |
| ③ 方案设计 | 2-3个方案比较·架构·核心逻辑 | 用户选中方案 |
| ④ 数据来源 | 数据从哪里来？精度？更新频率？ | 用户确认数据链 |
| ⑤ 审计+风险评估 | **用实际数据验证方案所有前提假设**（查库结构/查文件/查备份/查实际内容）→ 输出审计矩阵 → 确认无隐藏风险 | 用户确认 |

#### ⑤ 五维审计框架

当用户要求"评估审计"某个系统/方案时，按以下五维逐项输出审计矩阵：

| 维度 | 评估内容 | 检查方法 |
|:----|:---------|:---------|
| ① 逻辑自洽闭环 | 设计是否前后一致？有无悬空定义？步骤是否闭环？ | 读完整设计文档→标记不一致点→查对应实现 |
| ② 可行有效 | 设计是否有实际代码/文件/脚本落地？执行路径是否完整？ | 实际查文件存在性、脚本功能、cron注册 |
| ③ 效率提升 | 比之前做法快还是慢？设计上高效但实际呢？ | 有系统前/后对比，用数据说话 |
| ④ 解决断链 | 是否准确解决最初要解决的问题？覆盖范围够吗？ | 逐项对照原始需求→验证覆盖度 |
| ⑤ 死循环风险 | 有无自引/振荡/无限递归/无法终止的路径？ | 追踪每条循环路径→验证有无终止条件 |

**输出格式：** 带评分（绿/黄/红）的矩阵表 + 分数估算 + 核心发现摘要。不跳步，不夸大，诚实审计——发现设计vs代码差距直接说。

**每次只走一步**，用户确认后再走下一步。

## 文件结构

```
.discussions/                    ← 项目根目录
├── INDEX.md                     ← 全局索引（所有模块汇总）
├── {module}/                    ← 模块名（kebab-case）
│   ├── INDEX.md                 ← 本模块各次讨论摘要
│   ├── 2026-07-09.md            ← 按日记录
│   ├── 2026-07-10.md
│   └── ...
└── templates/
    └── session.md               ← 单次讨论记录模板（复制用）
```

## INDEX.md 格式

```markdown
# 讨论索引 — {模块名}

| 日期 | 进展 | 结论 |
|:----:|:----|:----|
| YYYY-MM-DD | 步骤①-③完成 | 核心结论 |
| YYYY-MM-DD | 步骤④-⑤完成 | 数据确认·待开发 |
```

## 讨论记录格式

创建新session时建议复制 `templates/session.md` 填入内容。格式如下：

```markdown
# {模块名} — 讨论记录

**日期**: YYYY-MM-DD
**阶段**: 步骤① 需求定义 / 步骤② 输入输出 / ...
**参与**: 用户 + Hermes Agent

---

## 步骤① 需求定义
- 目标描述
- 用户需求
- 结论

## 步骤② 输入输出
- 输入数据
- 输出结果
- 格式要求

...（按需续写，已完成步骤保留，新步骤追加）
```

## 行为规则

0. **"讨论:"强制触发**：用户消息以"讨论:"开头——立刻停掉手里的一切，进入讨论模式。不做任何例外判断。
1. **讨论不执行**：只产出方案文档，不写代码/不创建文件/不修改配置
2. **逐条确认**：每步结论必须用户确认后才进下一步
3. **二维索引**：先找模块→再找日期，不走捷径
4. **一次一模块**：不并行讨论多个模块，聚焦当前
5. **不重复**：已记录的结论不再问第二遍，直接引用
6. **一次一议题**：需要讨论多个事项时，每次只讨论一个。验证或确认这个议题的全部细节后，才能进入下一个。禁止一次性列出多个议题让用户选。用户明确要求"1个1个讨论"时立即切换至此模式。

## 与 system-improvement-framework 的关系

通用讨论 vs 系统迭代专用版：

| 场景 | 用哪个 | 原因 |
|:----|:------|:-----|
| 日常讨论（功能/流程/方案） | discussion-template | 5步通用框架 |
| 系统设计/迭代/重构/审计 | 两者叠加 | 此skill走5步，步骤③~⑤加载system-improvement-framework的五维审计+七步实施 |
| 用户说"用五维评估" | system-improvement-framework | 专用审计框架 |

**决策规则**：用户提"审计""评估""系统迭代""提升效率"→同时加载两个。system-improvement-framework是此skill在系统类议题下的专业扩展版。

## 陷阱

- **H001 讨论未记录**：讨论结束后忘记更新 `.discussions/{module}/YYYY-MM-DD.md` 和 INDEX.md。必须养成"讨论完立刻写"的习惯——用户说了"讨论："，结束时就一定同步更新文件。这条是最高频遗漏。
- **H002 跳步**：用户说"直接说方案"也先走需求定义
- **H003 结论不记录**：每步结束必须写结论到文件
- **H004 索引不同步**：每次更新 session.md 后同步更新 INDEX.md
- **H005 跑题**：当前步骤外的问题记"待定"不展开
- **H006 不执行**：讨论阶段只写 .discussions/ 下的记录文件
- **H007 假设数据源可访问**：讨论数据源时，必须用 curl/browser 实际验证网站的 HTTP 可访问性。许多中国网站（1688/B2B/造价信息网/招投标平台）有 WAF/验证码/企业CA证书等限制，理论可查≠实际可抓。先验证再定方案。
- **H008 一次性列多个议题**：讨论多个事项时禁止一股脑全列出来。每次只列一个，用户确认后再说下一个。用户说"你给建议"时也要先问优先级，逐一推进。
- **H009 讨论数据丢失时不检查备份就下结论**：当讨论涉及数据丢失/恢复场景时，必须先查已有备份体系再下结论——不能假设数据丢了。检查链：chat_backup.py每日消息JSON→state_weekly_backup.py每周全量state.db→auto_backup.py每日配置。确认完备份覆盖范围再判断到底丢了什么、能恢复什么。这条在用户系统里尤其重要，他有完善的3层备份架构。
- **H010 凭经验猜数据库结构而不实际查询**：当讨论涉及数据库方案（备份/清理/迁移/恢复）时，必须先用实际 SQL 查询验证表结构、行数、键值内容。不要凭经验猜测"应该有某某表"或"某字段应该是什么"。本次会话中助手错误猜测了 kv_store 和 alembic_version 两个不存在的表——直接降低用户信任。规则：做任何 DB 相关提案前，先 connect → sqlite_master → 行数 → 关键表内容，再说话。
- **H011 审计阶段跳过实际数据验证**：用户要求"审计"时必须走完整数据验证流程——查表结构、查行数、查关键列内容、对比新旧库差异。不能只做逻辑推演就下"安全"结论。审计的输出是一个有数据的矩阵（什么表/多少行/有什么风险/实际证据在哪），不是一句"我觉得没问题"。用户晏翔特别在意这个——错误假设会直接降低信任。
|- **H012 讨论目录路径歧义**：`.discussions/` 的存放位置依赖 Hermes 当前cwd。cwd不同导致讨论文件落在不同路径（本次BOM讨论就在 `WorkBuddy/Claw/.discussions/bom/` 而非 `~/.hermes/.discussions/bom/`）。规则：创建/加载讨论记录前，先确认cwd并统一使用绝对路径 `~/.hermes/.discussions/{module}/`。如果找不到标准路径下的讨论，主动扫描其他常见cwd位置（`WorkBuddy/Claw/`、`~`）下的 `.discussions/` 目录。
|- **H013 讨论产出skill未声明记忆层级**：每次讨论产出新skill或更新现有skill时，必须检查其是否声明记忆层级（`memory_tier`）。未声明的skill在后续使用时默认全量查询，导致噪音和漂移。规则：创建/更新skill时，在 frontmatter 中显式标注 `memory_tier: short-term | medium-term | long-term | auto`，并确保与其实际使用场景匹配。\n|- **H014 讨论内容不自动跨层检索**：讨论时引用历史结论只查 `.discussions/` 当前模块，忽略了其他模块或其他层级的记忆。规则：在讨论步骤③（方案设计）之前，先按3D记忆检索系统（state-db-lifecycle skill 七节）的fallback链快速扫描各层（state.db → pipeline/db → 其他模块讨论归档），确认当前话题有无过去已讨论过的类似方案或其他模块的相关经验。避免重复讨论已有结论或忽略跨模块关联。
- **H015 声明某组件不存在前未查全所有证据源**：审计时凭单一证据（如目录为空）即断言某系统"不存在"/"未实现"，但实际可能有脚本、cron、skill设计文档等其他证据表明它存在。规则：说某组件"不存在"前，必须逐项检查——目录存在？脚本存在？cron有注册？skill有文档？E盘有备份？全部查完才能下结论。2026-07-11审计memory-auto-tier时，助手因pipeline/目录为空就断言"管道库不存在"，但state_weekly_backup.py中明确有复制到pipeline/的逻辑，E盘也有两份全量备份。
- **H016 讨论自动化系统时止步于半自动方案**：当用户要求设计自动化系统时，"部分自动化+LLM手动补位"不是可接受的方案。用户晏翔明确指出：半自动=精神分裂。设计标准——系统主环必须100%自动（cron+脚本），LLM只做判断型决策（如"值不值得记"），不参与运行链。本次会话中记忆系统v1.0设计了20个操作但只有1个有代码→用户拒绝。v2.0改成4脚本+3cron全自动→通过。规则：出自动化方案时先问自己"这个系统中有没有一步需要LLM手动操作？"——有就重新设计直到没有。
