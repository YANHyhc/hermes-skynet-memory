# 系统迭代标准化流程 v1.0

> 源自 2026-07-11 记忆系统v2.0 讨论经验 — 从"20操作仅1落地"到"4脚本+3cron全自动"的成功路径

## 一、五维审计框架

任何系统迭代讨论开始前，用此五维框架做基线评估：

| 维度 | 含义 | 关键提问 |
|:----|:-----|:---------|
| 逻辑自洽闭环 | 设计是否逻辑一致？是否有缺口？ | 每个操作谁触发？谁执行？产出去哪？ |
| 可行有效 | 描述的说得通，实际跑得通吗？ | 去读文件！去查数据！去验证！ |
| 效率提升 | 相比当前做法是否真省了？ | 量化对比：之前几步/现在几步/token/时间 |
| 解决断链 | 是否解决了真正的问题？ | 断链的真实根因是什么？不是表面现象 |
| 死循环风险 | 会不会自己跟自己打架？ | 依赖有环吗？自引会触发吗？权重振荡吗？ |

**铁律：第②维"可行有效"必须查实际文件/代码/数据再下结论。严禁凭架构印象评价。**

## 二、七步迭代流程

> 从讨论到落地的完整闭环。每一步用户确认后再进下一步。

### 步骤① 需求定义
- 解决什么问题？谁用？期望产出？
- 输出：一句话目标描述
- 确认：用户说"对"

### 步骤② 现状摸底
- 查已有文件/代码/cron/数据，不凭记忆
- 输出：现状清单（有/无/能/不能）
- 陷阱：**自以为"没有"但实际存在** → 先查再说话

### 步骤③ 方案设计
- 写2-3个方案，不跳步
- 每个方案标注：LLM参与度 / 落地难度 / 自动程度
- 输出：方案对比表

### 步骤④ 方案审计
- 用五维框架审计方案
- 关键问题：
  - "不能做到全自动化？" → 追问到根因
  - "memory不支持分区？" → 变通方案是否有效？
  - "静态=蠢人系统？" → 是否解决了根本矛盾
- 输出：审计矩阵

### 步骤⑤ 代码落地
- 最小化原则：先做能自动跑的，再做锦上添花的
- 按依赖顺序实施（入口→生产→消费→维护）
- 每个脚本写完立刻验证

### 步骤⑥ 全链路验证
- 端到端跑通：触发→执行→产出→消费
- 不只看exit code 0，要看实际产出是否符合预期
- 清理测试数据

### 步骤⑦ 归档固化
- 更新相关skill
- 写入 `.discussions/{module}/`
- 更新assoc_index
- 输出五维终评（实施前→实施后对比）

## 三、关键提问模式

每次讨论中，以下问题能暴露最深层的设计缺陷：

| 问题 | 暴露什么 | 来自本次经验 |
|:----|:---------|:------------|
| "不能全自动？" | LLM参与度是否合理 | 记忆操作员=精神分裂 |
| "XX不支持XX？" | 变通方案是否解决根本 | memory不支持分区→skill映射解决 |
| "静态=死系统？" | 是否解决了根本矛盾 | 专注不流动=蠢人 |
| "分批分步都做了吗？" | 是否留下未完成项 | 第二批/第三批审计 |
| "用五维评估一下？" | 全面性检查 | 可行有效从35%→95% |
| "实际文件存在吗？" | 设计方案vs实际代码差距 | v1.0只有1/20有代码 |

## 四、实施铁律

1. **先查文件再说话** — 任何"没有"的结论必须亲眼看到目录为空
2. **只写能自动跑的代码** — 不写"概念代码"或"LLM自觉"的占位符
3. **每个脚本写完立刻验证** — 不假设成功
4. **先核心后锦上添花** — 第一批跑通CCCCC再考虑第二批
5. **讨论结束必须归档** — 不归档=白讨论
6. **最终产出必须用户看到** — 全链路验证结果展示给用户

## 七、陷阱

- H001 现状摸底凭经验不查文件 → 结论错误，丧失信任
- H002 方案设计跳步 → 用户说"直接说方案"也先走需求定义
- H003 审计阶段跳过实际数据验证 → 只有逻辑推演就下结论=无效审计
- H004 只写"概念代码" → 检查实现状态时发现1/20有代码
- H005 第二批/第三批做了一部分就报告全部完成 → 必须逐项审计
- H006 讨论结束不归档 → 关键结论丢失
- H007 不跑全链路直接汇报完成 → 自欺欺人
